# GoalTab Codebase Instructions

## Overview

**GoalTab** is a full-stack savings goal tracking application built with Vue 3 + TypeScript (frontend) and Express.js + MongoDB (backend). The app helps users create, track, and withdraw from savings goals with auto-save features, fulfillment options, and admin management.

---

## Architecture

### Frontend Stack

- **Vue 3** with `<script setup>` SFC syntax
- **TypeScript** for type safety
- **Vite** as build tool
- **Pinia** for state management with `pinia-plugin-persistedstate` for persistence
- **Tailwind CSS 4.0** for styling
- **Axios** with interceptors for API calls
- **Vue Router 4** for navigation
- **Capacitor 8** for mobile deployment (Android/iOS)

### Backend Stack

- **Express.js 5.2** with middleware pattern
- **MongoDB 9.2** with Mongoose ODM
- **JWT** authentication (Bearer token in Authorization header, fallback to `x-auth-token`)
- **Nodemailer** for email notifications
- **node-cron** for scheduled tasks (auto-save)
- **Paystack** integration for payments

## Frontend Patterns

### 1. API Client Pattern

**File:** `src/services/api.ts`

- Single axios instance with interceptors
- Auto-adds Bearer token from localStorage or Pinia store
- Handles 401 errors by clearing token and redirecting to /login
- Base URL from `VITE_API_URL` env or fallback to production

```typescript
api.interceptors.request.use((config) => {
  const token = localStorage.getItem("token") || authStore.token;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});
```

### 2. State Management (Pinia)

**Pattern:** Define-style stores with composition API

**Goals Store Example:**

```typescript
export const useGoalsStore = defineStore("goals", () => {
  const goals = ref<Goal[]>([]);
  const loading = ref(false);

  // Methods return promises and handle API calls
  const fetchGoals = async () => {
    loading.value = true;
    const { data } = await api.get("/goals");
    goals.value = data.map(normalizeGoal);
    loading.value = false;
  };

  // Computed for derived state
  const totalSaved = computed(() =>
    goals.value.reduce((sum, g) => sum + g.saved, 0),
  );

  return { goals, loading, fetchGoals, totalSaved };
});
```

**Conventions:**

- One store per domain (goals, auth, transactions, accounts, etc.)
- Use refs for mutable state, computed for derived values
- Methods handle API calls and update store state
- Normalize data from backend (convert `_id` to `id`)
- Use `storeToRefs()` in components to get reactive refs
- Persist state with `pinia-plugin-persistedstate` for offline support

### 3. Component Structure

**Vue 3 `<script setup>` SFC Pattern:**

```vue
<template>
  <div class="glass-card p-6">
    <h3 class="text-white">{{ goal.title }}</h3>
    <!-- Component markup -->
  </div>
</template>

<script setup lang="ts">
import { ref, computed } from "vue";
import type { Goal } from "@/types/goal";

interface Props {
  goal: Goal;
}

const props = defineProps<Props>();
const emit = defineEmits<{
  (e: "update-amount", value: number): void;
}>();

const localState = ref(0);

const derived = computed(() => {
  return props.goal.saved / props.goal.target;
});

const handleClick = () => {
  emit("update-amount", localState.value);
};
</script>
```

**Conventions:**

- Use TypeScript interfaces for props with `defineProps<Props>()`
- Type emits with `defineEmits<{}>()`
- Prefer composition hooks over mixins
- Component names match files (GoalCard.vue)
- Shared/reusable components in `components/shared/`
- Page components in `views/` (routed)

### 4. Tailwind Styling

- Use `glass-card` class for glass-morphism cards (semi-transparent dark)
- Color scheme: primary-500, gray-900 backgrounds, gray-400 text
- Spacing: standardized with Tailwind scale (4, 6, 8)
- Shadows: subtle, using `shadow-lg shadow-primary-500/20` patterns
- Light mode support with `light:` prefix (some components have theme toggle)

### 5. Composables Pattern

**Example:** `useOnboardingTour.ts`, `useDebouncedLoading.ts`

Reusable logic extracted into composables. Common patterns:

- Debouncing user input
- Loading state management
- Form validation (planned in useFormValidation.ts)
- Media queries (useMediaQuery)

### 6. Type Definitions

**File:** `src/types/goal.ts`, `src/types/user.ts`, `src/types/index.ts`

All domain models typed as interfaces. Key types:

- `Goal` — full goal object with metadata
- `GoalFormData` — form submission payload
- `Transaction` — transaction record
- `Withdrawal` — withdrawal request

## Backend Patterns

### 1. Route Structure

**File:** `backend/src/app.js`

Routes mounted with `/api/` prefix:

```javascript
app.use("/api/auth", authRoutes);
app.use("/api/goals", goalRoutes);
app.use("/api/withdrawals", withdrawalRoutes);
app.use("/api/admin", adminRoutes);
```

Each route file exports Express router with handlers.

### 2. Middleware Pattern

**Auth Middleware:** `backend/src/middleware/auth.js`

- Extracts token from:
  1. Authorization header (Bearer token)
  2. `x-auth-token` header
  3. Query param `?token=` (for SSE)
- Verifies JWT and attaches `req.user` object
- Returns 401 if missing or invalid

```javascript
const token =
  req.header("Authorization")?.slice(7) ||
  req.header("x-auth-token") ||
  req.query.token;
const decoded = jwt.verify(token, process.env.JWT_SECRET);
req.user = decoded.user;
next();
```

**Admin Middleware:** Checks if user has admin role

### 3. Mongoose Models

**Pattern:** Pre/post hooks for business logic

**Goal Model:**

- `userTarget` — user's stated amount
- `target` — userTarget + 100 (platform fee)
- `saved` — current savings
- `withdrawn` — total withdrawn
- `isClosed` — goal is finalized
- `progress` — calculated percentage
- `availableBalance` — virtual field (saved - withdrawn)

**Pre-save Hook:**

```javascript
GoalSchema.pre("save", async function () {
  // 1. Update progress
  this.progress = Math.round((this.saved / this.target) * 100);

  // 2. Deduct fee once when target reached
  if (!this.feeCharged && this.saved >= this.target && this.fee > 0) {
    this.saved -= this.fee;
    this.feeCharged = true;
  }

  // 3. Auto-close only when balance zero after withdrawal
  if (this.saved <= 0 && this.withdrawn > 0) {
    this.isClosed = true;
  }
});
```

### 4. Route Handler Pattern

**Convention:**

```javascript
router.post("/", auth, async (req, res) => {
  try {
    const { param1, param2 } = req.body;

    // Validation
    if (!param1) return res.status(400).json({ msg: "Missing param1" });

    // Lookup/authorization
    const item = await Model.findById(id);
    if (!item) return res.status(404).json({ msg: "Item not found" });
    if (item.user.toString() !== req.user.id) {
      return res.status(401).json({ msg: "Not authorized" });
    }

    // Business logic
    const result = await doSomething();

    // Create related records (logs, notifications, transactions)
    await Notification.create({...});
    await AdminLog.create({...});

    // Send email if needed
    await sendEmailToUser(req.user.id, "Subject", "<html>");

    // Response
    res.json(result);
  } catch (err) {
    console.error(err);
    res.status(500).json({ msg: "Server error" });
  }
});
```

**Conventions:**

- Use async/await
- Check auth at route level: `router.post("/", auth, ...)`
- Return early with status codes
- Populate related documents: `.populate("user", "name email")`
- Create audit logs and notifications for important actions
- Send emails for user-facing events

### 5. Withdrawal Flow

**Status:** `pending` → `approved` → `completed` (or `rejected`)

**Creation:** User submits with account details, **doesn't deduct immediately**
**Approval:** Admin approves, **then deducts from goal.saved** and adds to goal.withdrawn
**Validation:** Checks `availableBalance = saved - withdrawn`

### 6. Transaction & Activity Tracking

**Transaction Model:** Immutable record of money movement

```javascript
{
  (user, goal, type, amount, status, date);
}
```

**GoalActivity Model:** Detailed event log

```javascript
{
  (goal, user, type, amount, metadata, createdAt);
}
```

Types: `goal_created`, `funds_added`, `withdrawal_requested`, `auto_save`, etc.

## Common Workflows

### 1. Adding a New Feature

1. Create TypeScript type in `src/types/`
2. Create Pinia store in `src/stores/` (if needs state)
3. Create components in `src/components/[feature]/`
4. Create views in `src/views/` if routed page
5. Add routes in `src/router/`
6. Backend: Create model, routes, middleware as needed
7. Run `pnpm build` to type-check TypeScript

### 2. Creating a New API Endpoint

1. Create route in `backend/src/routes/[domain].js`
2. Add middleware (auth, admin) as needed
3. Export route and mount in `app.js`
4. Call from frontend via `api.get/post/put/delete()`
5. Handle response in Pinia store

### 3. Updating Goal Status

Goal status managed via fields:

- `progress` — calculated in pre-save hook
- `locked` — unlocked when goal completes
- `isClosed` — set when saved = 0 after withdrawal
- `fulfillmentStatus` — for product/service goals

## Key Domains

### Goals

- Create with `userTarget` (user amount) + auto-calculated `fee`
- Track with `saved`, `withdrawn`, `progress`, `locked`
- Support auto-save via cron jobs
- Platform fulfillment: product delivery or service booking

### Withdrawals

- User requests with bank details
- Admin approves/rejects
- Deduction happens on approval
- Status tracked: pending → approved → completed

### Transactions

- Immutable ledger of money movement
- Linked to goals
- Status: pending → completed/rejected
- Show in Recent Activity with status badges

### Authentication

- JWT tokens, stored in localStorage
- 24-hour expiry (check backend)
- Fallback header: `x-auth-token`

### Admin

- Approve/reject withdrawals
- View admin dashboard
- Access audit logs (AdminLog model)

### Frontend

- Pinia persistence for offline-first behavior

### Backend

- Index frequently queried fields (user, goal, status)
- Limit populated fields: `.populate("user", "name email")`
